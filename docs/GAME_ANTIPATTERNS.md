# 게임 추가 안티패턴 가이드 — "추가하지 말아야 할 것"

> 작성: 2026-06-12 · 분석 근거: git 커밋 이력(삭제 17건·수정 9건) + 사용자 피드백 메모리 + 디자인 규칙
> 용도: 새 프로젝트에 기존 게임 + 신규 게임을 **대량 추가**할 때, 과거에 거부·수정된 패턴을 반복하지 않기 위한 체크리스트.

이 문서는 두 종류의 실패를 다룬다.
- **A. 거부 패턴** — 어떤 *종류*의 게임이 만들어졌다가 사용자에게 삭제됐는가 (= 애초에 만들지 말 것)
- **B. 품질 안티패턴** — 게임 자체는 살아남았지만 만들 때 *반복해서 고쳐야 했던* 결함 (= 만들 때 처음부터 피할 것)

## 0.5 삭제 게임 전수 분석 — 5대 공통 축 (왜 잘렸나)

> 0절(커밋 이력 기반 삭제 17종) + `SELECTION.md`의 **사용자 직접 삭제 1~8차 38종**을
> 카테고리가 아니라 *왜 잘렸나*를 기준으로 재분류하면 아래 5개 축으로 수렴한다.
> 신규 발상 시 0절 목록뿐 아니라 **이 5개 축 중 하나에 해당하지 않는지** 먼저 자문할 것.

### 축① 기성 유명 퍼즐의 단순 이식 — puzzle 삭제의 핵심
독창성이 없고, 대개 **1인 정적 + 푸는 데 오래 걸림**이라 "30초·다인·왁자지껄" 콘셉트와 정면충돌.
> mini-sudoku(스도쿠) · logic-grid(노노그램) · car-escape(러시아워) · num-merge(2048) ·
> color-flood(Flood-It) · lights-out(라이츠아웃) · code-break(마스터마인드) · river-cross(강 건너기) ·
> frog-swap·peg-jump(고전 점프·페그 솔리테어) · flip-over(오델로/리버시) · word-search(낱말찾기)

### 축② 눈에 안 보이는 상태를 머릿속으로 추적·추론·세기
한눈에 안 들어오고 인지부하가 커서, 전자칠판 앞 다인 동시 플레이에 부적합.
> cup-shuffle(컵 위치) · flag-updown(청기백기 상태) · line-up(줄서기 추리) · gear-spin(톱니 방향) ·
> block-view·count-blocks·hidden-shapes(안 보이는·겹친 것 세기) · updown-number(숨은 수) ·
> rule-find(입출력 규칙) · clock-pass(시각 기억) · step-path(경로 추적) · maze-steps(칸 수 세기)

### 축③ 협력(coop)의 두 함정 — 말 설명 의존 / 얕은 분담
교실에서 말 전달은 느리고 주관적, 각자 한 부분만 맡는 협력은 판단이 얕다.
> **말로 설명·전달:** secret-code(색 순서) · coord-pass(좌표) · word-relay(음절) +
> assemble-shape·mirror-draw·clue-find(P1 설명→P2 맞히기)
> **얕은 분담·교대:** split-count(나눠 세기) · sort-together(공동 정렬) · sync-tap(동시 터치) ·
> fair-share(균등 분배) · turn-count(번갈아 누르기)

### 축④ 기존 게임과 본질이 같은 중복 클론
이름·소재만 다르고 *데이터 형식 + 인터랙션*이 동일.
> 짝찾기 find-pair-coop·twin-find·animal-buddy·attr-combo ↔ memory-match ·
> 텍스트 4지선다 synonym-quiz·capital-quiz·unit-noun · more-or-less↔size-compare ·
> coin-count↔dice-sum · spot-the-difference(두 그림 비교)

### 축⑤ 자제력·순간기억·찰나 반응
누르고 싶은 걸 참거나 찰나를 노리는 것 — 재미보다 답답함·좌절을 준다.
> countdown-tap(0에서 멈춤) · flash-shape(찰나 노출 기억) · emoji-zoom(점점 커지는 것 반응) · whack-a-mole(두더지 클론)

### 메타 통찰 — 삭제는 coop·puzzle에 쏠린다

사용자가 **직접 삭제 지시한 38종**(1~8차)의 카테고리 분포:

| 카테고리 | 직접 삭제 | 주된 축 |
|---|---:|---|
| 🧩 puzzle | 12 | 축① 기성 퍼즐 이식 |
| 🤝 coop | 11 | 축③ 말 설명·얕은 분담 |
| 🧠 brain | 8 | 축② 상태 추적·세기 |
| 🏃 speed | 3 | 축⑤ 참기·찰나 |
| ➗ math | 2 | |
| 📚 knowledge | 2 | |

→ **puzzle 12 + coop 11 = 전체의 약 61%.** 이 두 카테고리가 "교실·30초·다인 경쟁"이라는 제품
정체성과 **구조적으로 충돌하기 쉽다.** puzzle은 기성 퍼즐을 베끼면 1인 정적이 되고, coop은
말 전달·단순 분담으로 흐르기 쉽다. → **이 둘은 신규 발상 시 특히 신중할 것.**

> **한 줄 요약:** 잘린 게임은 거의 다 ―①기성 퍼즐 베끼기 ②머릿속 상태추적 ③말로 설명·얕은 분담
> ④기존과 중복 ⑤참기·찰나― 중 하나다. 공통적으로 *한눈에 안 들어오거나, 독창적이지 않거나,
> 30초 다인 경쟁에 안 맞는* 게임. 반대로 **살아남는 조건**은 ⓐ한 화면에 다 보이고 ⓑ여러 명이 동시에
> 경쟁하고 ⓒ객관적 정답이 즉시 판정되고 ⓓ30초 단발 — coop은 "양쪽 상시 활성+오답 페널티+명확한
> 데이터 조합", puzzle은 "다인 동시 경쟁+격자가 한눈에"여야 한다.

## A. 거부 패턴 (이런 *종류*는 만들지 말 것)

### A-1. 협력(coop) 게임의 함정 — 거부율 최고 (삭제 8/17)

협력 게임이 가장 많이 거부됐다. 살아남은 협력 게임과 대조하면 경계가 보인다.

**❌ 거부된 협력 메커니즘**
- **주관적 묘사 의존** — "P1이 그림/도형을 *말로 설명*하고 P2가 맞히기" (assemble-shape, mirror-draw, clue-find). 교실에서 말 설명은 주관적이고 느리고, 잘 안 통한다.
- **짝 찾기 협력** — find-pair-coop, twin-find, animal-buddy. 전부 memory-match/shape-match와 본질이 같다 (중복).
- **단순 번갈아 누르기** — turn-count. 자기 차례만 누르면 되니 "협력"이 아니라 그냥 교대.

**✅ 살아남은 협력 메커니즘 (참고 기준)**
secret-code·color-signal(명확한 **색 순서**를 전달), sum-relay(카드 **숫자 합**), jamo-merge(**자모** 조합), color-mix(**색** 섞기), word-chain(끝말잇기), direction-relay/memory-relay(**시퀀스**).

> **경계선:** 협력은 *객관적이고 명확한 데이터(색·숫자·자모·방향)*를 **조합하거나 전달**할 때 작동한다. *주관적인 그림·도형을 말로 묘사*하거나 *기존 게임과 겹치는 짝찾기*는 거부된다.

### A-2. 반응속도(speed) 게임의 함정 (삭제 6/17)

- **흔한 클론** — whack-a-mole(두더지). 너무 뻔하다.
- **자제력만 요구** — countdown-tap("0에 멈춤"). 누르고 싶은 걸 참는 게임은 재미보다 답답함.
- **순간 기억 의존** — flash-shape. 0.5초 노출 후 기억은 좌절감을 준다.
- **단조로운 단순 매칭** — color-switch, quick-sort, number-strike. "빠르게 누르기"만 있고 판단이 얕음.

### A-3. 중복 메커니즘 (카테고리 무관)

기존 게임과 *본질적으로 같은 플레이*는 카테고리만 바꿔도 거부된다.
- 짝 찾기 3종(coop) ↔ memory-match(brain)
- spot-the-difference ↔ "두 그림 비교" 자체가 반복적·제작 부담 큼

> **규칙:** 신규 발상 시 기존 게임 전체의 *데이터 형식 + 인터랙션*을 비교해, 이름만 다르고 플레이가 같으면 버린다.

## B. 품질 안티패턴 (만들 때 반복해서 고친 결함)

살아남은 게임들도 아래 결함으로 **사후 수정**이 필요했다. 처음부터 피하면 재작업이 없다.

### B-1. 난이도가 너무 낮음 ⚠️ (가장 잦은 수정)
- **증상:** turn-count(자기 차례만 활성→판단 불필요), direction-relay(25초→12초로 단축), 수 가르기(평면 무작위→3단계 점증)
- **규칙:**
  - 협력/순서 게임에서 **"자기 차례 버튼만 활성화"** 금지 → 항상 선택지를 다 열어두고 *오답에 페널티*. "지금 내 차례인가"를 판단하게 만들 것.
  - 제한시간은 넉넉하지 말 것. 라운드가 진행될수록 **점증 난이도**(시간 단축 또는 문제 복잡화).
  - 정답을 화면에 미리 표시하지 말 것 (turn-count는 다음 숫자를 버튼에 띄워서 암산이 불필요했음).

### B-2. 정답 모호성 / 오개념 유발 ⚠️
- **증상:** area-compare(SVG 자동 스케일로 "칸 수"가 아닌 "겉면적"이 커 보여 오답이 정답처럼), 단위 세기(한 물건이 두 단위로 다 맞아 보기 모호)
- **규칙:**
  - 정답이 **하나로 확정**되는지 검증. 보기 중 둘 이상이 맞을 수 있으면 그 보기를 제외하는 `avoid` 로직 필수.
  - 시각 비교 게임은 **렌더 크기가 정답을 왜곡하지 않는지** 확인 (셀 크기 고정 등). 오개념을 심으면 교육 게임으로 치명적.
  - 보기 오답은 정답 근처(±N)로 만들되 중복 없게.

### B-3. 시각 일관성 깨짐
- **증상:** nim-game(돌마다 랜덤 회색 음영 → 어색 / 버튼 안 돌이 `:first-child`만 색칠돼 1개만 다른 색)
- **규칙:** **같은 의미의 요소는 같은 색.** "다양성"을 위해 같은 종류 요소에 랜덤 색/음영을 주지 말 것. CSS 선택자는 `:first-child`가 아니라 전체(`circle[stroke]` 등)에 적용.

### B-4. 상태 스타일이 핵심 피드백을 가림
- **증상:** rhythm-echo(`disabled { opacity:0.5 }`가 `.flash` 강조 애니메이션을 가려 "소리만 나고 안 빛남")
- **규칙:** disabled/dim 같은 상태 스타일이 *강조·정답·하이라이트*를 덮지 않는지 확인. 강조 클래스에 `opacity:1 !important` + `z-index` 부여.

### B-5. 요소가 화면 밖으로 잘림
- **증상:** rps-react(답안 버튼이 220×70 비율 inline-SVG라 존 너비에 맞춰 세로로 커져, 보기 3개 중 2개만 보이고 3번째가 잘림)
- **규칙:** 답안/보기는 **고정 비율 SVG 대신 HTML/CSS 플렉스**로. `clamp()`로 화면 크기에 맞춰 축소되게. "보기 N개가 모두 한 화면에 들어오는지" 반드시 확인.

### B-6. 다인용(3인) 불공정 배치
- **증상:** 3인용에서 2+1 배치(P3만 하단 전체폭) → P3만 비대칭으로 불공정
- **규칙:** 3인용은 **3열 균등(1fr 1fr 1fr)**. 2+1 금지. 모든 플레이어 존이 같은 크기·같은 손 닿는 위치.

### B-7. 직접 그린 SVG가 어색함
- **증상:** rps-react(손 ✊✋✌️를 직접 SVG로 그렸더니 어색 → 시스템 이모지로 교체)
- **규칙:** 손·동물·복잡한 사물은 직접 그리기보다 **시스템 이모지**(`Apple Color Emoji` 등)가 더 명확하고 일관적. 직접 SVG는 단순 도형(원·사각·화살표)에 한정.
